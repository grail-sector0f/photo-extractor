# Phase 999.2: Settings View — Image Quality Control - Research

**Researched:** 2026-03-21
**Domain:** Chrome extension settings UI, chrome.storage.sync/local, React popup navigation
**Confidence:** HIGH

## Summary

This phase adds a settings screen where Jennifer can configure how the extension filters
images during a scan. The three control categories are: minimum resolution threshold
(skip images below NxN pixels), file type filtering (e.g. skip WebP or GIF), and CDN
upscaling preferences (enable/disable the rewrite behaviour per CDN provider).

The project already has the storage infrastructure, the content script dimension filter,
and the CDN rewrite module. This phase is about surfacing those existing constants as
user-editable preferences stored in `chrome.storage.local`, and adding a settings panel
to the popup UI that follows the MD3 design system already in use.

No new runtime dependencies are needed. The stack is already settled: WXT 0.20.x,
React 18, TypeScript, Tailwind CSS 3.x with MD3 color tokens, Vitest 3.x.

**Primary recommendation:** Store settings in `chrome.storage.local`, apply them at scan
time in the content script (minimum resolution) and at download time in the popup
(CDN enable/disable and file type filter). Expose settings via a slide-in panel or
second view within the existing 360x600 popup — a full separate entrypoint is overkill.

### Pattern 1: Settings Module (lib/settings.ts)

Store all defaults in one place. Every consumer imports from here — no magic numbers
scattered across files.

**What:** Pure TypeScript module defining the settings type, defaults, and
`chrome.storage.local` read/write helpers.

**When to use:** Any file that needs to read or write preferences.

**Example (to build):**
```typescript
// lib/settings.ts
export interface AppSettings {
  minDimension: number;       // default: 100 — skip images smaller than NxN
  allowedTypes: string[];     // default: ['jpg','jpeg','png','webp','gif','avif']
  cdnUpscalingEnabled: boolean; // default: true — apply CDN URL rewrites at download
}

export const DEFAULT_SETTINGS: AppSettings = {
  minDimension: 100,
  allowedTypes: ['jpg', 'jpeg', 'png', 'webp', 'gif', 'avif'],
  cdnUpscalingEnabled: true,
};

export async function loadSettings(): Promise<AppSettings> {
  const stored = await chrome.storage.local.get('settings');
  return { ...DEFAULT_SETTINGS, ...(stored.settings ?? {}) };
}

export async function saveSettings(settings: AppSettings): Promise<void> {
  await chrome.storage.local.set({ settings });
}
```

### Pattern 2: Pass Settings Through the Scan Message

The content script currently receives `{ type: 'SCAN_PAGE' }` from the popup and uses
the hardcoded `MIN_DIMENSION = 100` in both `imgTags.ts` and `content.ts`. The cleanest
way to make this user-configurable without adding a separate storage read inside the
content script (which would add async complexity at scan time) is to pass the relevant
settings in the scan initiation message.

**What:** Extend the `SCAN_PAGE` message payload to carry `minDimension`.

**When to use:** Any setting that affects what the content script scans.

**Example:**
```typescript
// In popup startScan():
port.postMessage({ type: 'SCAN_PAGE', minDimension: settings.minDimension });

// In content.ts handleScanSession():
const minDimension = msg.minDimension ?? 100;
// pass minDimension into extractImgTags() and processImg()
```

This keeps the content script stateless with respect to chrome.storage — the popup
owns the settings read and forwards what the content script needs.

### Pattern 3: Settings Panel in the Popup (not a separate entrypoint)

The popup is already 360x600px with a flex column layout. The settings screen should
replace the main view (not overlay it) using a simple boolean state flag: `showSettings`.
When `showSettings` is true, render a settings panel instead of the gallery/form. A back
button returns to the main view.

This is simpler than adding a second HTML entrypoint (which would require WXT entrypoint
boilerplate, separate state, and messaging between the two pages).

**What:** `showSettings: boolean` state in App.tsx (or a `view: 'main' | 'settings'` enum).

**Example:**
```typescript
// In App reducer or useState:
const [view, setView] = React.useState<'main' | 'settings'>('main');

// Render:
if (view === 'settings') return <SettingsPanel onBack={() => setView('main')} />;
return <MainView ... />;
```

### Pattern 4: CDN Upscaling Toggle

`cdnRewrite.ts` is a pure function (`rewriteUrlForMaxResolution`). The enable/disable
toggle is simplest to apply at the call site in `runDownloads()` in App.tsx:

```typescript
const upscaledUrl = settings.cdnUpscalingEnabled
  ? rewriteUrlForMaxResolution(url)
  : url;
```

No changes needed to `cdnRewrite.ts` itself.

### Pattern 5: File Type Filtering

`deriveExt()` in `naming.ts` already extracts the extension from a URL. The `allowedTypes`
setting is a filter applied to scan results before display (or at download time). Filtering
at scan time (content script) is cleaner because it prevents unwanted thumbnails from
appearing in the grid. Since extensions are determined from URLs, filtering can happen in
the popup after receiving `SCAN_RESULT` by comparing `deriveExt(image.url)` against
`settings.allowedTypes`.

This keeps the content script unchanged and applies filtering in the popup where settings
are already loaded.

### Anti-Patterns to Avoid

- **Separate popup entrypoint for settings:** WXT supports multiple HTML entrypoints, but
  a second popup page requires chrome.action.popup switching (complex) or a dedicated
  options page. The 600px popup is tall enough. Use view state instead.
- **Reading chrome.storage inside the content script for settings:** Adds async complexity
  and a race condition between storage read and scan execution. Pass settings via the
  scan message instead.
- **Hardcoding MIN_DIMENSION in multiple places:** It's already duplicated between
  `imgTags.ts` (line 28) and `content.ts` (line 41). This phase should consolidate to
  a single source of truth in `lib/settings.ts`.
- **Per-CDN toggles in v1:** Granular per-CDN on/off is overkill for Jennifer's use case.
  A single `cdnUpscalingEnabled` boolean covers the need. Keep it simple.

## Common Pitfalls

### Pitfall 1: MIN_DIMENSION Duplication
**What goes wrong:** `MIN_DIMENSION = 100` exists in both `lib/extract/imgTags.ts` and
`entrypoints/content.ts`. If the settings module introduces a third copy, the effective
threshold will diverge depending on which code path runs.
**Why it happens:** The constant was defined inline when each file was written.
**How to avoid:** In Plan 01, remove the hardcoded constants from both files and replace
them with the value received from `settings.minDimension` (passed through the scan message).
**Warning signs:** Tests for `imgTags.test.ts` and `mutationObserver.test.ts` that still
pass a hardcoded 100px threshold while the settings default has changed.

### Pitfall 2: Settings Load Timing in Popup
**What goes wrong:** The popup's `useEffect` for pre-fill and a new `useEffect` for
settings load both call `chrome.storage.local.get` on mount. If they run in the same
render cycle, React batches state updates but there are still two separate storage
reads, which is slightly wasteful.
**Why it happens:** Separate concerns (form pre-fill vs. settings) handled independently.
**How to avoid:** Merge both storage reads into a single `chrome.storage.local.get` call
on mount, reading all required keys at once.
**Warning signs:** Two separate `chrome.storage.local.get` calls inside `useEffect`.

### Pitfall 3: Settings Not Passed to Content Script
**What goes wrong:** Developer saves `minDimension: 200` to storage, but the content
script still uses its local `MIN_DIMENSION = 100` constant, so nothing changes.
**Why it happens:** Content script doesn't read from storage — it only receives messages.
**How to avoid:** Always pass `minDimension` in the `SCAN_PAGE` message payload and
remove the hardcoded constant from the content script entirely.
**Warning signs:** Changing the settings slider in the UI has no visible effect on scan results.

### Pitfall 4: chrome.storage.local.get Returns Undefined for Unset Keys
**What goes wrong:** On first install, `chrome.storage.local.get('settings')` returns `{}`
(not `{ settings: DEFAULT_SETTINGS }`), so `stored.settings` is `undefined`. Spreading
`undefined` over defaults throws or produces wrong output.
**Why it happens:** Chrome storage only returns keys that have been explicitly set.
**How to avoid:** Use `{ ...DEFAULT_SETTINGS, ...(stored.settings ?? {}) }` in `loadSettings`.
This is already the pattern used in the pre-fill logic.

### Pitfall 5: Popup Width Overflow from Settings Form
**What goes wrong:** The settings panel adds inputs and labels that overflow the 360px
popup width, causing horizontal scroll or clipping.
**Why it happens:** Inputs default to `width: auto` or `min-content`.
**How to avoid:** Use `w-full` on all inputs within the settings panel, consistent with
the existing `NamingForm` component.

## Open Questions

1. **What range should the minimum resolution slider cover?**
   - What we know: Current hardcode is 100px. Travel photos are typically 800px+. Icons/logos
     are typically under 50px. The UX-02 requirement says "e.g. smaller than 200x200".
   - What's unclear: Whether Jennifer wants a slider (continuous) or preset options
     (Small/Medium/Large thresholds).
   - Recommendation: Use a number input (min=50, max=500, step=50) with a default of 100.
     Simple, no slider component needed.

2. **Should file type filtering be per-type checkboxes or an allowlist text field?**
   - What we know: The current `KNOWN_EXTENSIONS` allowlist in `deriveExt()` is
     `['jpg','jpeg','png','webp','gif','avif']`.
   - What's unclear: Which types Jennifer actually wants to exclude in practice.
   - Recommendation: Start with a single toggle "Skip GIFs (usually icons/animations)"
     since GIF is the most common nuisance type. Full per-type checkboxes can come later.

3. **Should settings be scoped globally or per-hostname?**
   - What we know: Form pre-fill is already per-hostname (stored with `lastHostname`).
   - What's unclear: Whether Jennifer would want different min-resolution thresholds
     for different sites.
   - Recommendation: Global settings only for v1. Per-hostname settings add significant
     complexity for unclear benefit.
