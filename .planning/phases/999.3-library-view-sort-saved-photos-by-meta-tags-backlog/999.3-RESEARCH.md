# Phase 999.3: Library View — Sort Saved Photos by Meta Tags - Research

**Researched:** 2026-03-21
**Domain:** Chrome extension popup navigation, chrome.downloads API, chrome.storage.local as a data log, client-side sort/filter
**Confidence:** HIGH (architecture), MEDIUM (UX/scope details pending user input)

---

## Summary

Phase 999.3 wants a library screen where Jennifer can browse all previously saved photos and sort them by destination, vendor, category, and year.

The fundamental constraint is that **Chrome extensions cannot list the filesystem**. The extension cannot open `Downloads/travel-photos/` and enumerate files. It has two possible data sources: (1) `chrome.downloads.search()`, which queries Chrome's download history, or (2) a purpose-built download log written to `chrome.storage.local` at save time.

The download history approach sounds convenient but is brittle: users can clear browser history, Chrome purges old entries after a period, and `filenameRegex` searches are substring-based (requiring careful filtering). Parsing metadata back out of filenames is also fragile — the `notes` field can contain underscores, making reliable splitting impossible for some filenames.

A download log stored in `chrome.storage.local` is the correct approach. It is written by the extension itself, never touched by the browser's clear-history action, persists indefinitely, and stores metadata as structured fields (no parsing required). The tradeoff is that photos saved before this phase ships will not appear in the library.

**Primary recommendation:** Write a structured download log to `chrome.storage.local` at save time. Render the library as a third view (`'library'`) in the existing `view` state pattern established in Phase 999.2.

## Critical Design Question: Data Source

This is the most consequential decision for this phase. It affects what's possible, how reliable the library is, and what backfill story (if any) exists for older saves. It should be discussed with the user before planning begins.

### Option A: Parse chrome.downloads history

- Query `chrome.downloads.search({ filenameRegex: 'travel-photos' })` to get items saved by the extension.
- Parse the filename to extract destination, vendor, category, year, notes.
- Sort/filter the parsed results.

**Pros:** Requires no change to the existing save pipeline; shows historically saved photos immediately on launch.

**Cons:**
- Chrome download history is user-clearable and may be purged automatically.
- `filenameRegex` is a substring search; you must filter carefully to avoid matching other files.
- Filename parsing is unreliable. The format `destination_vendor_category_year_notes_index.ext` uses underscore as separator, but `notes` can contain hyphens only (not underscores — normalizeField strips non `[a-z0-9-]` chars). That makes parsing deterministic IF you know how many fixed fields came before notes. But `year` and `notes` are both optional, making split-by-underscore unreliable without knowing which segments are which.
- `DownloadItem.filename` is the absolute path (e.g. `/Users/jennifer/Downloads/travel-photos/bali_four-seasons_pool_2025_01.jpg`). The relative portion must be extracted.
- `chrome.downloads` history is only accessible from the background service worker (or via a message to it).

### Option B: Write a download log to chrome.storage.local (RECOMMENDED)

At save time (inside `runDownloads`, after each successful download), write a structured record to `chrome.storage.local`:

```typescript
interface SavedPhotoRecord {
  id: string;          // uuid or timestamp-based — unique per save
  url: string;         // original source URL
  filename: string;    // relative path: "travel-photos/bali_four-seasons_pool_2025_01.jpg"
  destination: string; // raw (pre-normalize) or normalized — normalized is fine
  vendor: string;
  category: string;
  year: string;
  notes: string;
  savedAt: string;     // ISO 8601 timestamp
}
```

Store as an array under key `'savedPhotos'` in `chrome.storage.local`. Append on each save. Cap array at a reasonable limit (e.g. 500 records) with an LRU eviction policy to avoid hitting the 10 MB storage quota.

**Pros:**
- Metadata stored as structured fields — no parsing.
- Immune to browser clear-history.
- Fast reads — no downloads.search() call needed.
- Thumbnail URLs available for display.

**Cons:**
- Photos saved before this phase ships will not appear in the library.
- Adds a write at save time (minor overhead, negligible in practice).
- `chrome.storage.local` has a 10 MB quota by default (can be lifted with `unlimitedStorage` permission, but that's rarely needed for text records).

**Backfill option (optional):** On first library open, if the log is empty, also query `chrome.downloads.search()` and attempt a best-effort parse of filenames. Show a notice that only partial history is available. This is optional complexity — discuss with user.

### Pattern 1: Library Data Layer (lib/library.ts)

Mirrors the pattern in `lib/settings.ts`: a pure module with typed helpers and no side effects on import.

```typescript
// Source: mirrors settings.ts pattern in this codebase
export interface SavedPhotoRecord {
  id: string;
  url: string;          // source URL (for thumbnail display)
  filename: string;     // relative path "travel-photos/bali_..."
  destination: string;
  vendor: string;
  category: string;
  year: string;
  notes: string;
  savedAt: string;      // ISO 8601
}

const STORAGE_KEY = 'savedPhotos';
const MAX_RECORDS = 500;

export async function loadLibrary(): Promise<SavedPhotoRecord[]> {
  const data = await chrome.storage.local.get(STORAGE_KEY);
  return (data[STORAGE_KEY] as SavedPhotoRecord[]) ?? [];
}

export async function appendToLibrary(record: SavedPhotoRecord): Promise<void> {
  const existing = await loadLibrary();
  const updated = [...existing, record];
  // LRU cap: if over limit, drop oldest entries
  const capped = updated.length > MAX_RECORDS
    ? updated.slice(updated.length - MAX_RECORDS)
    : updated;
  await chrome.storage.local.set({ [STORAGE_KEY]: capped });
}
```

### Pattern 2: Wiring appendToLibrary into runDownloads

In `App.tsx`, inside `runDownloads`, after each successful `sendDownloadMessage` call:

```typescript
// After each successful download, append a record to the library log
```

### Pattern 3: LibraryPanel Component

```typescript
// Follows the SettingsPanel component pattern in App.tsx

  onBack: () => void;
}

  const [records, setRecords] = useState<SavedPhotoRecord[]>([]);

  const [filterTag, setFilterTag] = useState<string>('');

  useEffect(() => {
    loadLibrary().then((r) => setRecords([...r].reverse())); // newest first
  }, []);

  // Client-side sort
  const sorted = [...records].sort((a, b) => {
    if (sortBy === 'savedAt') return b.savedAt.localeCompare(a.savedAt);
    return a[sortBy].localeCompare(b[sortBy]);

  const filtered = filterTag
    ? sorted.filter(r =>
        r.destination.includes(filterTag) ||
        r.vendor.includes(filterTag) ||
        r.category.includes(filterTag) ||
        r.year.includes(filterTag)
      )
    : sorted;

  return (
    // sticky header with back arrow + "Library" title (mirrors SettingsPanel)
    // scrollable list of filtered/sorted records
    // each record shows: thumbnail img, filename, destination/vendor/category/year tags
  );
}
```

### Anti-Patterns to Avoid

- **Don't parse filenames to recover metadata.** The notes field can contain hyphens, making reliable parsing fragile. Store metadata as structured fields at write time.
- **Don't use chrome.downloads.search() as the primary data source.** History is user-clearable and access requires a background message round-trip.
- **Don't add a new background message type.** The popup can read `chrome.storage.local` directly — no background involvement needed for reads.
- **Don't try to display a thumbnail by reading the local file.** The `url` field in `SavedPhotoRecord` is the original source URL from the web page. Use that for the thumbnail `<img>` src. It may be stale (the page may 404 later), but it is the best available option without filesystem access.

| Problem | Don't Build | Use Instead | Why |
| --- | --- | --- | --- |

| Sort/filter | A sort library | Native Array.sort + Array.filter | The dataset is tiny (< 500 records); no performance concern |
| Persistent storage | IndexedDB | chrome.storage.local | Already in use in this project; simpler API; sufficient for 500 text records |

**Key insight:** The library dataset will rarely exceed a few hundred records for a solo travel advisor. No specialized data structures, indexing, or query engines are warranted.

## Common Pitfalls

### Pitfall 1: Thumbnail URLs Go Stale

**What goes wrong:** The `url` stored in `SavedPhotoRecord` is the page's original image URL. After some time, that URL may return a 404 (CDN cache expired, page taken down, etc.). The `<img>` tag will break.

**Why it happens:** The extension saves URLs at download time, not the downloaded file content.

**How to avoid:** Handle `onError` on the thumbnail `<img>` the same way `ThumbnailCard` already does — hide the broken image and show a placeholder `?` div. Don't try to re-download or resolve the URL.

**Warning signs:** Library renders but some thumbnails show the placeholder.

### Pitfall 2: Records Accumulate Duplicates on Re-Download

**What goes wrong:** If Jennifer rescans a page and re-downloads the same photo (e.g., same URL), the library log will contain duplicate entries for that URL.

**Why it happens:** `appendToLibrary` appends unconditionally.

**How to avoid:** Deduplicate by URL on append — check if a record with the same `url` already exists before adding. Or accept duplicates and note it as a known limitation. Either is acceptable; discuss with user.

### Pitfall 3: chrome.storage.local Quota

**What goes wrong:** Default quota is 10 MB. Text records are tiny, but storing long URLs for 500+ records could theoretically approach the limit if notes are long.

**Why it happens:** chrome.storage.local has a per-extension 10 MB quota by default.

**How to avoid:** The 500-record LRU cap keeps storage well within limits. A single record with URL, filename, and fields is roughly 200–500 bytes. 500 records ≈ 100–250 KB. No risk of hitting the quota.

**Warning signs:** `chrome.storage.local.set()` rejects with a quota error.

### Pitfall 4: runDownloads is in App.tsx, Not Easily Testable

**What goes wrong:** `runDownloads` is a module-level function in App.tsx that imports React internals indirectly. Adding `appendToLibrary` there adds side effects to an already-complex function.

**Why it happens:** `runDownloads` was written as a standalone function but lives inside the popup component file.

**How to avoid:** The existing pattern works fine. Just mock `appendToLibrary` in tests the same way `sendDownloadMessage` is tested (spy on the chrome.storage.local.set call). No structural change to runDownloads is needed.

### Pitfall 5: Popup Cannot Access chrome.downloads from Its Own Context

**What goes wrong:** If the library needs to cross-reference with download history (e.g., check `exists` field), it must go through the background worker.

**Why it happens:** `chrome.downloads` is only available in the background service worker in MV3.

**How to avoid:** Do not use `chrome.downloads` in the library at all. The download log in `chrome.storage.local` is the only data source the popup reads.

```typescript
// Source: codebase pattern (lib/settings.ts loadSettings/saveSettings)
export async function appendToLibrary(record: SavedPhotoRecord): Promise<void> {
  const data = await chrome.storage.local.get(STORAGE_KEY);
  const existing: SavedPhotoRecord[] = (data[STORAGE_KEY] as SavedPhotoRecord[]) ?? [];

  // (Optional: discuss with user whether duplicates should be allowed)
  const withoutDupe = existing.filter(r => r.url !== record.url);

    ? updated.slice(updated.length - MAX_RECORDS)
    : updated;

}
```

## Open Questions

1. **Backfill for existing saves**
  - What we know: Photos saved before this phase ships have no log entry.
  - What's unclear: Does Jennifer want to see those historical downloads in the library, or is "starts fresh from when the library ships" acceptable?
  - Recommendation: Start fresh and add a one-time notice on first library open: "Your library shows photos saved after [date]. Earlier saves aren't included." This avoids the complexity of filename parsing.

2. **Duplicate handling**
  - What we know: If Jennifer re-downloads a photo from the same URL, appendToLibrary will see the same URL.
  - What's unclear: Should the same URL/destination/session produce one record or multiple? Is re-downloading intentional (different trips, different context)?
  - Recommendation: Allow duplicates by default. The user controls this via normal file management. The library is a log, not a deduplicated index.

3. **Library display format**
  - What we know: Records have destination, vendor, category, year, notes, and a thumbnail URL.
  - What's unclear: Does Jennifer want a card grid (thumbnails prominent) or a table/list (metadata prominent)? Sort by date vs. alphabetical?
  - Recommendation: Default to newest-first list with thumbnail, filename, and tag chips. Let the plan phase define the UI spec in more detail.

4. **Delete from library**
  - What we know: The extension cannot delete files from the filesystem. The log record can be removed from chrome.storage.local.
  - What's unclear: Should there be a delete-from-log option? This would hide the record from the library but the file still exists on disk.
  - Recommendation: Start without delete functionality. It can be added later.
